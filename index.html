DP Payments — visibility in search scoring

Focus: payment-related score contributions only · Snapshot: 13 Apr 2026 · Markets: Brazil, Spain, Italy, Mexico, Poland

Scope of this note

This page describes how DP Payments signals can change search positioning (extra score when a signal is on). It does not explain the rest of the search algorithm or other non-payment boosts.

Scoring reads the same marketplace payment index flags used elsewhere; rules at booking time can still differ from what the index records.

Five payment signals that can add score

When a signal is true for an indexed address, it can contribute its weight toward the blended score — alongside many non-payment ingredients. More than one payment signal can be true at once.

  • Online payments on for that address (secure-payments path in the index).
  • Frequently paid flag (stronger “momentum” signal in the index).
  • Commercial tier — three levels; base tier adds no tier score.

Payment tiers 0–3 (what they are)

Tiers are buckets of counted marketplace payment activity per address. We refresh that picture on a rolling basis. Tiers 1–3 unlock the tier part of the search boost; tier 0 means no tier boost.

Poland uses paid-visit style counts; other active markets use paid transactions above a minimum value — still within a recent window.

Facilities — location

Here the bar moves with how many featured doctors are attached to the same address: a busier clinic needs more payments to reach tier 2 or 3 than a small one. If there are no featured doctors or no payments, the location stays at tier 0.

Poland — “frequently paid” and tiers
Only in Poland, tier 2 or 3 can also switch on the extra frequently paid boost and patient-facing line. For specialists that applies on their own practice listings; for facilities, whenever the location reaches tier 2 or 3. Other markets don’t use tiers to drive that second signal.

Where in the backend this is wired

In monolith-app, the five payment score hooks are registered only on the recommended sorter in app/config/misc/search_bridge.yml. They are not listed under the other sorter ids in that same file.

Whether a given patient session actually uses that sorter is a product/UI question — this document only states what the backend attaches to the recommended profile.

Verified in repo
  • sorters → recommended → factors includes payments_has_badge, payments_frequently_paid, payments_tier_1, payments_tier_2, payments_tier_3.
  • Those five identifiers do not appear under other sorters keys in that YAML file.
  • Gate GATE_PAYMENTS_BADGES_ON_SEARCH_RESULTS in GateKeeper.class.php: br, es, it, mx, pl — when off, the payment factor classes contribute nothing.
