DP Payments — visibility in search scoring

Focus: payment-related score contributions only · Snapshot: 13 Apr 2026 · Markets: Brazil, Spain, Italy, Mexico, Poland

Scope of this note

This page describes how DP Payments signals can change search positioning (extra score when a signal is on). It does not explain the rest of the search algorithm or other non-payment boosts.

Scoring reads the same marketplace payment index flags used elsewhere; rules at booking time can still differ from what the index records.

What can add payment-related score

Each item below can contribute its weight when the condition is met — alongside many non-payment ingredients. Several can be true at once (e.g. high activity tier + Poland “popular” hook).

  • Online payments on for that address.
  • Activity tiers 1–3 — volume bands; each tier adds its own weight once crossed.
  • Poland only — “popular specialist / facility” — not a separate volume story: the same tier 2 / tier 3 activity unlocks the second patient-facing line and an extra registered score line (see weights table).

Payment tiers 1–3 (activity bands)

Tiers are buckets of counted marketplace payment activity per address. We refresh that picture on a rolling basis. Each of tiers 1–3 adds its own slice of the search boost once volume crosses that band.

Poland uses paid-visit style counts; other active markets use paid transactions above a minimum value — still within a recent window.

Facilities — location

Here the bar moves with how many featured doctors are attached to the same address: a busier clinic needs more payments to reach tier 2 or 3 than a small one. With no featured doctors or no payments, there is no tier boost for that location.

Poland — “popular specialist / facility” line & extra score
In Poland only, reaching activity tier 2 or 3 also switches on one index flag used for (a) the “popular specialist / facility” patient line and (b) the extra score row named payments_frequently_paid in config — not a separate volume tier; it follows the same high bands as tiers 2–3. Tier 1 alone does not trigger it. Doctors vs facilities: for specialists, the code blocks this on cross-listed appearances so the line stays on their own practice listing; facilities use Poland + tier 2/3 without that extra rule. Other markets keep tier weights but do not set this Poland flag from tiers.

Markets & entity shape

Payment scoring (and related indexing) is gated to: Brazil, Spain, Italy, Mexico, Poland — same five for specialists and facilities.

Relative weights (search_factor table)

Poland DB checked; names repeat per country. Higher = stronger contribution when that line is on. Tiers are the volume ladder; the last row is the Poland-only add-on tied to tier 2 or 3 (badge + messaging), not a separate activity bucket.

How to read this (verified in monolith): For visit pay-ins, CreateMarketplaceVisitPaymentPayinCommandHandler always sets doctor_id from the visit’s doctor. It sets facility_id from address.facility_id for the visit address (may be NULL), and crm_customer_id from the visit’s marketplace payment account. This table sums pay-ins whose account owner is a facility crm_customer in the cohort, so it still counts visits booked to a doctor when the address row has no facility_id but the money sits on the facility’s customer. It is not “patient chose facility with no doctor.”
