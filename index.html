DP Payments in search — what patients see & when it moves the needle

One-pager for stakeholders · Operational data snapshot: 13 Apr 2026 · Five active markets: Brazil, Spain, Italy, Mexico, Poland

In one minute

In patient search we can show short trust cues linked to paying through Docplanner (e.g. “secure payments”, and where data supports it a “patients like paying online here” style message). Separately, the same underlying flags can add a small scoring bonus in search — but only inside the “Recommended” ranking recipe defined in our backend (see verification box below). They are not part of the simpler “sort by rating” recipe in that same configuration.

“Any payment signal” = at least one address row with online payments on, or “frequently paid” messaging flag, or a non-base commercial tier. Figures from production-style DBs; definitions in the tables further down.

What appears on the card (patient view)

  • Secure payments badge — when that location is marked as having online payments enabled in our search index. (Shown only in markets where the feature gate is on.)
  • “Renowned” / popularity line — when a separate “frequently paid” flag is set; wording differs for an individual specialist vs a facility.
Why finance / support should care
The badge reflects account + address level setup in the index, not “this exact service has a fixed online price”. In edge cases the badge can show while checkout still blocks payment (e.g. only “price from” services). Booking-time rules are stricter than the search badge.

Markets & who it applies to

Feature gate on only for: Brazil, Spain, Italy, Mexico, Poland. Same list for specialists and for facilities — there is no second country matrix per entity type.

Specialists

Signals are stored per doctor practice address. The system can also pick up related “promoted” doctor links when rebuilding the index.

Facilities

Signals are stored per facility location (facility address), tied to that facility’s payment profile.

Search ranking — “Recommended” vs “Rating”

Backend configuration (app/config/misc/search_bridge.yml in monolith-app) lists many ingredients for the recommended sorter — including the five payment-related boosts. The rating sorter in that same file only lists rating and raw_score; none of the payment boosts appear there.

Other sorters in that file (suggested, search_assistant_priority, etc.) also do not include the payment boosts. We did not re-check here which option the patient app selects by default — only what the backend attaches to each named sorter.

Checked in code (no guesswork)
  • Where payment boosts sit: only under sorters → recommended → factors — lines that register payments_has_badge, payments_frequently_paid, payments_tier_1, payments_tier_2, payments_tier_3. File: monolith-app/app/config/misc/search_bridge.yml.
  • Where they do not sit: sorters → rating → factors lists only rating and raw_score in that file.
  • Gate for boosts + indexing: factor classes return nothing when GATE_PAYMENTS_BADGES_ON_SEARCH_RESULTS is off — GateKeeper.class.php enables that gate for br, es, it, mx, pl.
